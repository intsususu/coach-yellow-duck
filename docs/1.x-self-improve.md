# HealthApp 1.x 自查与改进清单

> 本文是在通读现有代码后整理的技术改进清单。目标：在进入 2.0 大功能（见 [2.0planning.md](2.0planning.md)）之前，先把 1.x 的正确性 bug、卡顿与架构债清掉，否则会被带进 2.0 放大。
>
> **本清单按「重要度」排序**（综合：用户可感知程度 × 紧迫性 ÷ 改动成本），每条标注问题类型【正确性 / 性能 / 架构 / 健壮性 / 可维护】：
> - **P0 = 立即修**：用户可感知，且改动小，可在下次发版一并带上。
> - **P1 = 架构地基**：阻碍 2.0，必须在大功能开工前做。
> - **P2 = 随手做**：健壮性与次要性能，配合对应 2.0 功能开发时顺手处理。

---

## P0 · 立即修（用户可感知 + 改动小）

### P0-1 【正确性】目标体重 `goalWeight` 不持久化，重启丢失
- **现象**：在「我的」改完目标体重，重启 App 后回到默认 73.0。
- **根因**：[AppState.swift:39](HealthApp/App/AppState.swift:39) `@Published var goalWeight: Double = 73.0` 只有内存默认值；ProfileView 通过 [ProfileView.swift:57](HealthApp/Features/Profile/ProfileView.swift:57) `appState.goalWeight = newGoal` 写入，但**没有 didSet 落盘、init 也不读取**。
- **影响**：目标线、「距目标」、综合分析的「距目标」全部基于错误值。
- **修法**：给 `goalWeight` 加 `didSet { userDefaults.set(...) }`，init 时读取（已有注入的 `userDefaults`，几行即可）。进 2.0 持久化迁移时并入设置存储（见 P1-1）。

### P0-2 【正确性】事件叠加开关 `showsEvents` 不持久化
- **根因**：[AppState.swift:45](HealthApp/App/AppState.swift:45) `@Published var showsEvents = true`，同样无落盘/读取。
- **影响**：用户关掉趋势图事件叠加，重启又回到开启。
- **修法**：同 P0-1，归入统一设置存储。

### P0-3 【性能 · 卡顿】图表渲染路径里反复 `new DateFormatter()`（首要卡顿嫌疑）
- **现象**：`DateFormatter` 的创建出了名的昂贵（每次约毫秒级）。多个图表把它放在**每帧、每个坐标轴标签都会调用**的函数里现造：
  - [WeightChart.swift:254](HealthApp/Charts/WeightChart.swift) `axisLabel(for:)` —— 每个 X 轴刻度标签都 new 一个，滚动时每帧若干次。
  - [SleepChart.swift:571](HealthApp/Charts/SleepChart.swift)、[ExerciseChart.swift:67](HealthApp/Charts/ExerciseChart.swift) 与 [ExerciseChart.swift:411](HealthApp/Charts/ExerciseChart.swift) 同样模式。
  - 列表/卡片侧也有：[EventTimelineView.swift:287](HealthApp/Features/Events/EventTimelineView.swift)、[RecentEventRow.swift:41](HealthApp/Features/Home/Components/RecentEventRow.swift)、[AnalysisViews.swift:438](HealthApp/Features/Analysis/AnalysisViews.swift)、[HomeView.swift:298](HealthApp/Features/Home/HomeView.swift)。
- **对照**：[WeightView.swift:343](HealthApp/Features/Weight/WeightView.swift) 已用 `static let recordDateFormatter`——这正是正确做法，图表/其它文件没沿用。
- **影响**：横滑趋势图时持续创建/销毁 formatter，主线程抖动 → 掉帧。
- **修法**：所有 formatter 改为 `static let` 复用（按 range/格式各留一个），或集中到一个格式化工具。**改动小、收益最直接，建议第一个做。**

### P0-4 【性能 · 卡顿】图表计算属性在单次 body 内反复重排序 / 重过滤
- **现象**：[WeightChart.swift:21](HealthApp/Charts/WeightChart.swift) `sortedSamples` 是**计算属性**，每次访问都 `samples.sorted{...}`（O(n log n)）。而它在一次 body 求值里被 `visibleWindow`、`trailingPadSeconds`、`visibleEvents`、`yDomain`、主 `ForEach`、`markerEvents`、`weightOnTrend` 等**多处反复访问**，每处都重排一遍。
- **放大效应**：图表绑定了 `chartScrollPosition`，**滚动时 body 每帧重算**，于是每帧排序 N 遍。`yDomain`/`visibleWindow`/`visibleEvents` 也都是链式重算的计算属性，叠加过滤开销。SleepChart / ExerciseChart 有相同结构。
- **影响**：数据点越多、滑动越久越明显，正是"偶尔卡一下"的典型成因。
- **修法**：把排序后的数组、窗口、域等**算一次**（在 ViewModel 里预排序，或用 `let` 在 body 起始处求值并传递；样本已知有序时直接去掉 `.sorted`）。

### P0-5 【体验诚信】「iCloud 同步」开关是死按钮
- **根因**：[UserProfile.swift:51](HealthApp/Features/Profile/UserProfile.swift:51) 注释明说"当前仅记录偏好，iCloud 同步后续开放"。UI 能切 `local / iCloud`，但不触发任何同步。
- **影响**：误导用户以为已开启 iCloud（数据安全相关，误导成本高）。
- **修法**：在 2.0 真正接上 iCloud（见 [2.0planning.md](2.0planning.md) 功能 1）前，先把开关**置灰并标注「即将开放」**。**在真正实现前，不应让它看起来可用。**

## P1 · 架构地基（阻碍 2.0，大功能前必须做）

### P1-1 【架构】持久化层迁移（关键路径 · 2.0 多功能的地基）

> 这是进入 [2.0planning.md](2.0planning.md) 多数功能（iCloud/日历/体脂/控件/私密）前的**前置技术工程**。功能 1/2/3/4/7 全部依赖它，越晚迁移、历史数据越多，迁移成本越高。P0-1/P0-2 的设置持久化应作为本迁移的一部分正式落地，而非长期停留在临时补丁。

**现状问题**：用户数据高度碎片化，散落在至少 4 处、3 种机制，且无单一持久化抽象：

| 数据 | 位置 | 代码 |
|---|---|---|
| 事件 | UserDefaults 单 key JSON | [EventStore.swift](HealthApp/Repository/EventStore.swift) |
| 用户资料 / 存储偏好 | UserDefaults JSON + string | [UserProfile.swift:89](HealthApp/Features/Profile/UserProfile.swift) |
| 头像图片 | Documents 目录 | [UserProfile.swift:145](HealthApp/Features/Profile/UserProfile.swift) |
| 趋势快照 | Application Support JSON | [CachingHealthRepository.swift:26](HealthApp/Repository/CachingHealthRepository.swift) |
| goalWeight / showsEvents | （应在 UserDefaults，实际未存，见 P0） | [AppState.swift](HealthApp/App/AppState.swift) |

这套方案无法满足 2.0：
- iCloud 同步（UserDefaults 不跨设备，KV Store 仅 1MB）
- Widget 读取（extension 读不到主 App 的 UserDefaults）
- 大容量 / 结构化查询（日历按天聚合、体测记录、私密事件加密）

**目标存储**：**SwiftData / Core Data + CloudKit（`NSPersistentCloudKitContainer`）**，容器放在 **App Group 共享目录**。
- 事件、体测记录走 CloudKit 实体；
- 轻量设置（goalWeight、showsEvents、主题）走 `NSUbiquitousKeyValueStore`；
- HealthKit 指标不入库（本身跨设备）。

**迁移要点**：
1. entitlements 增补 iCloud（CloudKit + KV Store）与 App Group capability。
2. 把 `EventStore` 抽象为协议，新增 SwiftData 实现，保留旧实现读取旧数据。
3. 写一次性"UserDefaults JSON → SwiftData"的搬运迁移（复用现有 `HealthEvent` Codable 迁移经验），迁移完成打标记避免重复。
4. `AppState.events` 单一数据源接口尽量不变，把改动收敛在仓库层，减少各页改动面。

**本迁移顺带解决的两处历史小债**：
- **设置散用裸字符串 key**【可维护】：多处直接 `userDefaults.set(_, forKey: "字符串")`（AppState、ProfileStore 各自定义 Key）。收敛为一个 `Settings` 类型（或 `@AppStorage` 包装），集中管理 key 并为 KV Store 留接口。
- **事件保存双重写入**【性能】：[AppState.saveEvent](HealthApp/App/AppState.swift:181) 先改内存数组，再 `repository.saveEvent` → `EventStore.upsert` 内部又 `load()` 整体反序列化、再整体序列化落盘，单条保存触发一次全量读写（量大后随总数线性增长）。迁到 SwiftData 后天然按记录写入，问题自然消解。

**验收**：事件在两台设备间自动同步；Widget 能读到事件；旧版本升级后历史事件不丢；目标体重 / 事件叠加开关重启后保留。

### P1-2 【质量】零自动化测试，纯逻辑模块未被覆盖
- **现状**：仓库内无测试 target / 测试文件（仅有构建产物日志）。
- **风险点**：以下都是**纯函数、易测、且后续会被 2.0 复用**的关键逻辑，一旦回归无人发现：
  - `AnalysisReportEngine`（[AnalysisReport.swift:82](HealthApp/Features/Analysis/AnalysisReport.swift)）：体重变化、频率、情绪判定、文案分支。
  - `aggregateSleep` 跨午夜归属（[HealthKitRepository.swift:543](HealthApp/Repository/HealthKitRepository.swift)）：2.0 日历(功能 2)按天聚合会直接依赖它的口径。
  - `EventStore` upsert/delete/迁移、`HealthEvent` 旧数据 Codable 迁移。
- **行动**：建一个单元测试 target，先覆盖 `AnalysisReportEngine` 和 `aggregateSleep`（用构造样本，不依赖真机 HealthKit）。投入小、收益高，也为 P1-1 持久化重构兜底。

## P2 · 随手做（健壮性与次要性能）

### P2-1 【健壮性】HealthKit 错误被静默吞掉，无可观测性
- [HealthKitRepository.swift](HealthApp/Repository/HealthKitRepository.swift) 中几乎每个查询都是 `catch { return [] }`，错误既不上报也不打日志。真机上「为什么没数据」将无从排查。
- **行动**：引入 `os.Logger`，在 catch 分支记录错误（不打扰 UI，仅便于诊断）。2.0 接大模型分析(功能 5)和推送(功能 6)后，可观测性更重要。

### P2-2 【性能】事件保存触发四个 Tab 全量重绘
- [AppState.saveEvent](HealthApp/App/AppState.swift:181) 用 `events = updatedEvents` **整体替换** `@Published` 数组；首页/体重/睡眠/运动四个 tab 都读 `appState.events`，一次保存令四页连同其图表事件叠加全部重算。
- **影响**：单次记录事件后短暂卡顿；与 P0-4 叠加（每页图表又各自重排）会更明显。
- **行动**：先做 P0-4 后影响大幅下降；长期由 P1-1 持久化重构 + 更细粒度状态拆分解决。

### P2-3 【性能】启动期主线程同步 I/O 与图片处理
- 冷启动在主线程同步做了几件事，叠加最短 splash 1.4s（[AppState.swift:36](HealthApp/App/AppState.swift)）：
  - [CachingHealthRepository 初始化](HealthApp/Repository/CachingHealthRepository.swift:82) 同步读盘 + JSON 解码趋势快照；
  - [ProfileStore 初始化](HealthApp/Features/Profile/UserProfile.swift:108) 同步读头像文件并跑 `dominantColor()` 吸色（48×48 像素遍历，主线程）。
- **影响**：单次启动延迟，不是滑动卡顿，但影响"打开 App 即时感"。
- **行动**：快照/头像读取与吸色移到后台，就绪后回主线程赋值；吸色结果可缓存避免每次启动重算。

### P2-4 【健壮性】综合分析加载一次后不刷新、各指标取数范围不一致
- [AnalysisViewModel.swift:22](HealthApp/Features/Analysis/AnalysisViewModel.swift) `guard weights.isEmpty && ...` 一旦加载就不再刷新；新记录的体重/事件不会反映到再次打开的分析里。
- 同时 weights 用 `.month`、sleeps 用 `.month`、workouts 全量，三者取数窗口口径不一（虽底层都拉得够长，但语义混乱）。
- **行动**：提供显式刷新入口或基于数据指纹失效缓存；统一分析取数范围常量。

### P2-5 【可维护】硬编码常量散落
- 睡眠目标写死 `sleepGoalHours: 8`（[HealthKitRepository.swift:33](HealthApp/Repository/HealthKitRepository.swift)）；
- 体重「全部」起始年写死 `2019`（[HealthKitRepository.swift:394](HealthApp/Repository/HealthKitRepository.swift)）；
- 活动环兜底目标 30 分钟 / 500 千卡（[HealthKitRepository.swift:307](HealthApp/Repository/HealthKitRepository.swift)）。
- **行动**：睡眠目标应可由用户设置（与 goalWeight 同归设置层）；其余抽为具名常量，便于 2.0 个性化。

### P2-6 【性能】五个 Tab 常驻 + 各自 `.task` 加载
- `TabView`（[MainTabView.swift:11](HealthApp/App/MainTabView.swift)）首次切到某 tab 即实例化并常驻，每页 `.task` 各自发起仓库查询。虽有 `CachingHealthRepository` 预热与去重兜底，但缓存未命中（如下拉刷新后）时多页并发查询仍可能争用主线程。
- **影响**：次要；缓存命中时无感。
- **行动**：依赖现有缓存层即可；如需进一步，可在切 tab 时按需加载、对重计算结果做记忆化。

### P2-7 【健壮性】少量强制下标 / 兜底缺失
- [UserProfile.swift:146](HealthApp/Features/Profile/UserProfile.swift) `FileManager.default.urls(...)[0]` 直接取下标。正常设备不会为空，但属可避免的强制访问，迁移持久化层时一并改为安全解包。

## 建议执行顺序

1. **先清 P0**（goalWeight / showsEvents 持久化、iCloud 开关置灰；P0-3 formatter 静态化、P0-4 图表去重排序）——全部用户可感知、改动小，可一并立即发版。**卡顿优先从 P0-3 / P0-4 入手。**
2. **建测试 target（P1-2）覆盖纯逻辑** —— 为后续重构兜底。
3. **进入 2.0 第 0 阶段（P1-1）**：统一持久化层（顺带解决散用 key、事件双写），把 P0 的设置持久化并入正式存储方案。
4. **P2 随对应 2.0 功能顺手做**：P2-1 配合大模型/推送、P2-5 配合个性化设置、P2-3 配合启动优化、P2-4 配合分析页迭代。
