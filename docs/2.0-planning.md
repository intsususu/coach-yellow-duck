# HealthApp 2.0 规划概述

> 本文是 2.0 阶段的功能发散与落地规划，基于现有架构（SwiftUI + HealthKit）撰写。
> 现有关键设计：
> - **数据分两层**：健康指标（体重/睡眠/运动）来自 HealthKit；**事件**走本机持久化（`EventStore` → UserDefaults + JSON）。
> - **单一数据源**：`AppState.events` 为全局事件源，各页只读叠加，写入只在事件模块。
> - 已具备版本迁移意识（`HealthEvent` 的 Codable 兼容旧数据）、缓存层（`CachingHealthRepository`）、综合分析模块（`AnalysisReport`）。

---

## 目标功能清单

1. 接入 iCloud，保存用户设置信息与事件信息
2. 日历功能：在「我的」页以日历视角查看每日总结状态，点击查看详情
3. 加入体脂肪 / 体脂率类指标，支持 InBody 体测数据截图、照片 OCR 识别
4. 加入小控件（Widget）：自律打卡、事件记录、日历展示
5. 综合分析接入 AI 大模型 API
6. 推送功能：仅每周总结，克制，不加其他推送
7. 事件记录增加私密事件（生物指纹识别后可记录和展示）

> **前置技术工程**：本规划的多个功能（iCloud/日历/体脂/控件/私密）都依赖一次统一的**持久化层迁移**。该迁移属技术改进，单独记在 [1.x-self-improve.md](1.x-self-improve.md) 的「持久化层迁移」一节，是进入下列功能前的关键路径。

---

## 1. iCloud 同步（设置 + 事件）

**核心障碍**：事件当前存在 UserDefaults，不适合同步。

两条路：
- **轻量路（改动小）**：`NSUbiquitousKeyValueStore`（iCloud KV，上限 1MB / 1024 key）。现有 `EventStore` 是"一个 key 存一坨 JSON"，几乎可平移。**但 1MB 上限**——一旦第 3 点加入 InBody 截图/照片就会爆。
- **正路（推荐）**：事件迁到 **SwiftData / Core Data + CloudKit**（`NSPersistentCloudKitContainer`）。一次性改造 `EventStore`，换来自动同步、冲突合并、容量无忧，也为第 2、3、7 点打地基。

**建议**：后续日历(2)、私密事件(7)、体脂(3)都围绕事件/记录扩展，**先做持久化层迁移**，否则每个新功能都在 UserDefaults 上叠债。

**坑**：
- 设置类（goalWeight、showsEvents、主题）适合 KV Store；事件 / 体测记录适合 CloudKit。**两者分开**，别混。
- HealthKit 数据**不要**自己同步——它本身跨设备同步，只同步"事件 + 设置 + 体测录入"。
- 需在 entitlements 加 iCloud capability（已有 `HealthApp.entitlements`，需补 CloudKit + KV）。
- 写一次性"UserDefaults → 新存储"的搬运迁移，复用已有 Codable 迁移经验。

## 2. 日历视图（「我的」页，每日总结状态 + 点击详情）

与现有架构契合度最高，**第一个适合做的可见功能**。

- **每日"状态"来源**：可聚合四类数据成一天的"健康分/状态点"——体重达标、睡眠时长、运动消耗、是否有特殊事件。建议定义 `DaySummary`（某天指标快照 + 事件列表 + 一个状态色）。
- **视觉**：月历每格一个**状态点/环**（复用 `Color+Tokens` 事件色板与首页 ring 思路）。有事件的日子叠事件色点，纯数据日子按健康分着色。
- **点击详情**：弹出当天 detail——指标 + 当天事件，复用 `RecentEventRow`、`AnalysisReport` 小结文案。
- **发散**：月历可加"连续打卡"高亮，与第 4 点小控件、自律打卡天然联动。
- **坑**：HealthKit 按天聚合注意时区与"睡眠归属哪一天"（凌晨睡眠算前一天还是当天，沿用 Sleep 模块现有口径）。

## 3. 体脂 / 体脂率（InBody 截图 OCR）

最有挑战、也最出彩。**分两阶段**。

**阶段 A（先做，价值大风险小）**：手动录入体脂率 / 体脂肪 / 骨骼肌量字段。
- HealthKit **原生支持** `bodyFatPercentage` 和 `leanBodyMass`——**优先写入/读取 HealthKit**，与体重一样自动跨设备同步（顺手解决第 1 点对这类数据的同步）。
- 数据模型仿 `WeightSample` 做 `BodyCompositionSample`，图表复用 `WeightChart` 套路。

**阶段 B（OCR）**：截图/拍照识别。
- 优先用 **Apple 原生 Vision（`VNRecognizeTextRequest`）**——离线、免费、隐私好，InBody 单据是规整印刷数字，识别率高。**别一上来就上大模型 OCR。**
- 流程：选图 → Vision 抽文本 → 正则/关键字定位"体脂率/骨骼肌/BMI"等字段 → **进确认页让用户核对再保存**（OCR 必须有人工确认环节，别静默写入）。
- **进阶**：识别不稳的版式可把 OCR 原始文本丢给第 5 点大模型做结构化提取——让 3 和 5 协同。

**坑**：InBody 有多种机型/报告版式，正则会脆；先支持自己那台版式，再逐步加。

## 4. 小控件 WidgetKit（打卡 / 事件记录 / 日历）

- **技术前提**：Widget 是独立 extension，**不能直接读主 App 的 UserDefaults**——必须走 **App Group 共享容器**。再次指向第 1 点：**先把存储迁到 App Group 下的共享存储（或 CloudKit），Widget 才拿得到数据。**
- 三种控件按成本排序：
  - **日历/状态展示**（只读，最简单）：展示本周打卡点或今日健康环。
  - **自律打卡**（中等）：iOS 17+ 用 **App Intents + interactive widget**，控件上直接点"已打卡"。
  - **快速记事件**（较难）：控件点击只能深链回 App 打开编辑器（控件里塞完整表单不现实）。
- **额外**：iOS 16+ 锁屏小组件、iOS 18 控制中心控件，均可作为"一键记录/打卡"入口。
- **优先级**：放在日历(2)与打卡机制成型后再做，否则没数据可展示。

## 5. 综合分析接入大模型 API

已有 `AnalysisReport` / `AnalysisViewModel`——"分析框架已就位，只差把规则引擎换成/补成 LLM"。

- **架构建议**：保留现有本地规则分析作为**兜底**，LLM 作为**增强**。无网络/超时/失败 → 回落到现有报告。健康类 App 不能因断网就没分析。
- **调用方式**：
  - **不要把 API key 打进客户端**。正规做法经自己的轻量后端中转。个人/自用 App 可本地存 key，但要讲清风险。
  - 用 **Claude API**（最新 Opus 4.8 / Haiku 4.5）。把"指标趋势 + 本周事件 + 目标"组织成结构化 prompt，产出本周点评 + 建议。周总结低频场景用 Opus 质量好；省成本可用 Haiku。
- **隐私**：上传健康数据，**务必脱敏 + 明确告知 + 可开关**。只传聚合趋势数字，不传原始截图/私密事件（与第 7 点冲突，私密事件绝不入 prompt）。

## 6. 推送（仅每周总结，克制）

- **关键决策：本地通知 vs 远程推送。**
  - 只需"每周固定时间提醒看本周总结"——**用本地通知（`UNUserNotificationCenter`）即可**，无需服务端、无需 APNs 证书、零后端成本。**推荐。**
  - 远程推送只在"内容要服务端实时生成/跨设备触发"时才需要——克制定位用不上。
- **设计**：周日晚固定时间本地通知 →"本周总结已就绪"→ 点击深链进第 5 点分析报告。
- **若**希望"推送里直接带 AI 生成的一句话总结"，才需后台生成 + 远程推送，但违背"克制"初衷，建议不做。
- **坑**：需请求通知权限（可并入现有 A3 引导风格）。

---

## 7. 私密事件（生物识别后记录/展示）

- **认证**：`LocalAuthentication`（`LAContext`，Face ID / Touch ID，失败回落设备密码）。
- **数据模型**：给 `HealthEvent` 加 `isPrivate: Bool`（已有成熟 Codable 迁移模式，新字段 `decodeIfPresent` 默认 false，零迁移痛点）。
- **关键体验决策**：
  - 列表/日历/图表中，私密事件**默认隐藏或打码**（显示"🔒 私密事件"占位），认证通过后才显示内容。
  - **想清楚"叠加图表"**：私密事件是否参与趋势叠加与综合分析？建议——**参与统计但不显示内容**（图上显示锁标记的点），否则记了等于没记。
- **安全坑（重点）**：
  - 生物识别只是"门"，**数据本身要加密**。私密事件内容应存 **Keychain** 或用 `CryptoKit` 加密后落库，不能明文躺在 UserDefaults/数据库——否则越狱/备份可直接读出，生物识别形同虚设。
  - 私密事件**绝不能**进第 5 点 LLM prompt、不能进第 1 点非加密同步、不能在第 4 点 Widget 明文展示。
  - `LAContext` 要处理"设备没设生物识别"的回落。

## 建议的迭代顺序

功能间有强依赖，顺序很重要：

| 阶段 | 做什么 | 为什么先做 |
|---|---|---|
| **0（地基）** | **持久化层迁移**（技术工程，详见 [1.x-self-improve.md](1.x-self-improve.md)）：事件从 UserDefaults → SwiftData/CloudKit + App Group | 1/2/3/4/7 全都依赖它，越晚迁债越重 |
| **1** | iCloud 同步(1) + 日历视图(2) | 地基铺好后顺势完成，日历是高性价比可见功能 |
| **2** | 体脂手动录入→写 HealthKit(3A) + 私密事件(7) | 都是事件/指标模型的自然扩展 |
| **3** | 大模型分析(5) + 每周本地通知(6) | 5 给 6 提供内容，一起做 |
| **4** | OCR(3B) + 小控件(4) | 最复杂，依赖前面数据成型 |

---

## 两个最重要的提醒

1. **持久化迁移是真正的关键路径**（技术细节见 [1.x-self-improve.md](1.x-self-improve.md)）。当前 5 个功能（iCloud/日历/体脂/控件/私密）都卡在"UserDefaults 存事件"这个点上。建议下一个迭代先做它，别急着做单点功能。
2. **体脂数据(3)优先用 HealthKit 原生类型**，能白嫖跨设备同步，别自建。
